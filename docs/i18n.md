# i18n Structure

Open Model Lab uses `next-intl` with locale-prefixed public routes and a canonical-English content model.

- Supported locales live in `i18n/routing.ts`.
- Shared UI messages live in `messages/<locale>.json`.
- Canonical English content lives in `content/catalog/*` and `content/concepts/*`.
- Optional optimized English overlays live in `content/optimized/concepts/*`.
- Locale content overlays live in `content/i18n/<locale>/...`.
- `pnpm content:registry` regenerates the runtime bundle at `lib/i18n/generated/content-bundle.ts`.
- `pnpm content:registry` also regenerates the concept variant manifest at `content/_meta/generated/concept-variant-manifest.json`.
- `pnpm content:registry` also regenerates the optimized concept UI-copy bundle at `content/_meta/generated/optimized-concept-ui-copy.json`.

## Route And Message Model

- Public URLs are locale-prefixed, for example `/en/...` and `/zh-HK/...`.
- The locale route segment is the authoritative locale source of truth.
- `app/[locale]/layout.tsx` mounts `NextIntlClientProvider` keyed by the current locale so client-side locale changes remount cleanly.
- `i18n/request.ts` loads locale messages on top of English, so missing message keys fall back to `messages/en.json`.

Shared reusable chrome belongs in message catalogs:

- header/footer/navigation labels
- shared CTA, loading, error, and empty states
- concept-page framework labels
- pricing/contact/account chrome
- simulation shell labels

## Content Overlay Model

English remains the canonical source of truth. Locale overlays are partial, sharded, and safe to merge over English. Optimized English uses the same partial-overlay model and is resolved before canonical English.

Preferred source layout:

- `content/i18n/<locale>/catalog/subjects.json`
- `content/i18n/<locale>/catalog/topics.json`
- `content/i18n/<locale>/catalog/starterTracks.json`
- `content/i18n/<locale>/catalog/guidedCollections.json`
- `content/i18n/<locale>/catalog/recommendedGoalPaths.json`
- `content/i18n/<locale>/concepts/<slug>.json`

Runtime loading:

- `lib/i18n/content.ts` never reads the filesystem directly and stays lightweight for shared display helpers.
- `lib/i18n/concept-content.ts` owns rich concept variant resolution for full concept bodies and should stay on server/test code paths.
- `scripts/generate-i18n-content-bundle.mjs` reads the checked-in shards, migrates any legacy monoliths if needed, and emits `content/i18n/generated/<locale>.json`.
- `scripts/generate-content-variant-bundle.mjs` reads canonical English, optimized overlays, locale overlays, and editorial metadata, then emits the generated optimized bundle plus variant manifest under `content/_meta/generated/`.
- `tools/i18n/common.py` must stay aligned with `lib/content/editorial-overlays.mjs` for concept overlay field allowlists and source-hash computation; update them together when the editorial contract changes.
- `lib/i18n/generated/content-bundle.ts` imports those generated locale bundles for runtime-safe use in the app.
- `lib/content/generated/content-variants.ts` imports the generated optimized bundle and variant manifest for runtime-safe use in the app.
- `lib/content/generated/content-variant-ui.ts` imports the lightweight optimized concept UI-copy bundle for card/listing surfaces.
- Missing localized fields fall back to optimized English when available, otherwise canonical English.

Concept fallback order:

- `en`: optimized English -> canonical English
- non-`en`: localized -> optimized English -> canonical English

Merge rules:

- Canonical ids, slugs, progress keys, analytics keys, and share keys never change.
- Only allowlisted user-facing editorial fields are mergeable. Protected structural fields are ignored and reported during generation.
- Arrays of translated objects merge by stable `id` where available.
- When object arrays depend on identity-based merges, locale overlays must keep those stable `id` or `slug` keys instead of pruning them away.
- Primitive arrays are treated as full replacements, so translated arrays should keep canonical shape.
- Overlay keys outside the allowlisted editorial structure are stripped and reported in status output; structural shape mismatches on allowed fields still fail validation.
- Localized manifests also record whether missing locale fields fell back to optimized or canonical English and expose fallback counts and sampled field paths.
- Empty `{}` optimized overlays are treated as no-op placeholders and skipped cleanly.

Catalog parity:

- Concept card/listing helpers already use localized concept UI copy and optimized English UI copy where available.
- Subject/topic/starter-track/guided/goal-path catalogs continue to localize through `content/i18n/<locale>/catalog/*`.
- Optimized English overlays are intentionally concept-only for now.

## Translation Tooling

The content-translation pipeline lives under `tools/i18n/`.

Primary entry points:

- `python tools/i18n/translate_content.py --locale zh-HK`
- `python tools/i18n/validate_overlays.py --locale zh-HK`

Convenience package scripts:

- `pnpm i18n:translate -- --locale zh-HK`
- `pnpm i18n:validate -- --locale zh-HK`
- `pnpm i18n:worklist -- --locale zh-HK`
- `pnpm i18n:check:zh-HK`
- `pnpm i18n:sweep:zh-HK`

Optional compatibility wrappers remain available:

- `python tools/i18n/translate.py --locale zh-HK`
- `python tools/i18n/validate.py --locale zh-HK`

### Translate Command

`translate_content.py` is schema-aware and writes overlay shards rather than mutating canonical English.

Supported modes:

- source file mode: `python tools/i18n/translate_content.py content/concepts/simple-harmonic-motion.json --locale zh-HK`
- source directory mode: `python tools/i18n/translate_content.py content/concepts --locale zh-HK`
- full-content mode: `python tools/i18n/translate_content.py content --locale zh-HK`
- single-concept / batch filtering: repeat `--concept <slug>`
- dry-run planning: add `--dry-run`
- changed-only mode: default behavior, or spell it explicitly with `--changed-only`
- resume mode: add `--resume` to retry only previously attempted items that are failed, incomplete, or stale, while skipping current/done items and untouched items
- force full rerun of the selected set: add `--all` or `--force`
- alternate overlay destination for review/smoke work: add `--overlay-root <path>`

Selection behavior:

- Default `--changed-only`: translate selected items that are new, failed, missing, or stale; skip current done items.
- `--resume`: narrow work to previously attempted items that still need attention. This retries failed entries, stale entries, and incomplete entries, but skips current done items and skips untouched items that have no manifest entry yet.
- `--all`: disable changed-only filtering and process the full selected set.
- `--force`: force translation for the full selected set even if those items are current.
- `--all` or `--force` take precedence over `--resume`, so combining them reruns the full selected set.

Real-provider payload contract:

- The CLI never asks the model to echo encoded JSON path keys such as `["sections","explanation","paragraphs","0"]`.
- Each provider request chunk gets stable short aliases such as `f0001`, `f0002`, and the alias-to-path map stays local to the tooling layer.
- The provider request stays intentionally simple: human-readable source context plus alias-keyed `fields`.
- The provider response may include known wrapper metadata keys such as `itemId`, `itemKind`, `sourceTitle`, or `context`; those are ignored before alias validation.
- The provider response must still be a single JSON object whose field aliases exactly match the expected alias set for that chunk.
- The CLI maps aliases back to canonical encoded paths only after strict validation succeeds.
- Single-field token repairs still return alias-only JSON. Required markers and marker-to-original maps are passed only as retry context and never become accepted output keys.

Chunking behavior:

- Real runs split larger items into multiple provider requests to reduce local-model drift.
- If a chunk fails because of alias omission, unexpected aliases, wrapper drift, or token mismatch, the translator retries that same chunk with a stricter repair prompt.
- If the chunk still fails, the translator splits only the failing chunk into smaller chunks, then retries again.
- If needed, the fallback continues down to single-field retries before the task is marked failed.
- Token-heavy single-field retries carry an explicit required-marker list and marker-to-original map, and they get one final token-focused repair attempt before the field is marked as a hard failure.
- Successful chunk results are kept and merged deterministically before the overlay shard is written.
- Default limits are `--max-fields-per-request 24` and `--max-chars-per-request 2400`.
- Either limit can be disabled with `0`, but the normal Ollama path should keep both enabled.

Provider selection:

- `--provider mock` is deterministic and test-safe.
- `--provider ollama` uses a local Ollama server through its OpenAI-compatible `/v1` endpoint.
- `--provider` defaults to `ollama` for real translation runs.
- `--provider openai-compatible` remains optional and non-default for future custom endpoints; normal repo workflows should use local Ollama.

Relevant env vars:

- `OPEN_MODEL_LAB_I18N_PROVIDER`
- `OPEN_MODEL_LAB_I18N_OLLAMA_BASE_URL`
- `OPEN_MODEL_LAB_I18N_OLLAMA_MODEL`
- `OPEN_MODEL_LAB_I18N_OLLAMA_TIMEOUT_MS`
- `OPEN_MODEL_LAB_I18N_OLLAMA_RETRIES`
- `OPEN_MODEL_LAB_I18N_OLLAMA_TEMPERATURE`

No cloud API key is required for the default translation workflow. The standard path is local-only and keeps translation traffic on the developer machine.

Local-only workflow:

- Install Ollama locally from [ollama.com](https://ollama.com/) if it is not already available.
- Start Ollama locally, for example with `ollama serve`.
- Pull a model you want to use, for example `ollama pull llama3.2`.
- Run translations with `pnpm i18n:translate -- --provider ollama --model llama3.2 --locale zh-HK`.
- Run a provider health check with `pnpm i18n:ping -- --provider ollama --model llama3.2`.

Recommended rollout for a real translation wave:

1. `pnpm i18n:ping -- --provider ollama --model <local-model> --locale <locale>`
2. `pnpm i18n:translate -- content/concepts/<one-concept>.json --provider ollama --model <local-model> --locale <locale> --overlay-root <temp-dir>`
3. `pnpm i18n:translate -- content/concepts --provider ollama --model <local-model> --locale <locale> --concept <small-batch-slug> ...`
4. Review the generated shards before widening the batch.
5. `pnpm i18n:validate -- --locale <locale>`

CLI overrides:

- `--provider ollama`
- `--base-url http://127.0.0.1:11434/v1`
- `--model llama3.2`
- `--timeout-ms 120000`
- `--retries 2`
- `--temperature 0`
- `--max-fields-per-request 24`
- `--max-chars-per-request 2400`

The web app runtime does not import any translation provider code or depend on Ollama at runtime. Ollama is tooling-only.
Translation jobs send machine identifiers plus human-readable source context to the provider. Concepts use the canonical concept title, and catalog runs use human-readable collection labels such as `Subjects catalog`.
Mock runs are intentionally cleaner than real LLM runs. Real Ollama runs use alias-keyed chunks, strict JSON validation, and retry-on-repair logic because local models can drift or return malformed keys even when a mock server behaves perfectly.
Protected tokens are masked with compact, copy-stable markers before translation, for example `[[M0]]`, `[[P1]]`, `[[I2]]`, `[[R3]]`, and `[[S4]]`. Token preservation is count-based rather than order-sensitive, so reordered protected tokens pass if the same protected token multiset is preserved, but missing or invented protected tokens still fail.

### zh-HK Fallback Worklist Export

Use the worklist export when a locale overlay is structurally valid but still falls back to English at runtime.

- `pnpm i18n:worklist -- --locale zh-HK`
- `pnpm i18n:worklist -- --locale zh-HK --bucket 51+`
- `pnpm i18n:worklist -- --locale zh-HK --slug simple-harmonic-motion`
- `pnpm i18n:worklist -- --locale zh-HK --min-fallback-count 20`

The command writes generated artifacts under `content/_meta/generated/`:

- `i18n-worklist-<locale>.json`
- `i18n-worklist-<locale>.md`
- filtered runs add deterministic suffixes such as `--bucket-51plus` or `--slug-simple-harmonic-motion`

Each JSON work item contains:

- the concept slug, title, locale, target overlay path, fallback count, and full fallback field paths
- the runtime English base in use for those missing fields (`optimized` or `original`)
- a minimal `sourcePatch` containing only the untranslated editorial fields that are still falling back

`sourcePatch` is a valid partial overlay, not a full concept file. It keeps stable `id` / `slug` anchors where needed for array merges, but it does not include protected config fields such as simulation params, formulas, URLs, or other structural branches.

Use the exported `sourcePatch` as the copy/paste payload for manual review or an external LLM workflow. Editors should translate only the values, then merge the translated patch into `content/i18n/<locale>/concepts/<slug>.json`.

Do not alter placeholders, LaTeX, math tokens, or other protected strings inside the exported fields. Preserve them exactly.

After updating locale overlays, rerun:

1. `pnpm content:registry`
2. `pnpm content:variants:status`
3. `pnpm content:variants:validate`
4. `pnpm i18n:validate -- --locale zh-HK`

### Validation Command

`validate_overlays.py` checks:

- overlay JSON parses correctly
- overlay files correspond to real canonical concepts/catalog collections
- overlay keys stay inside the allowlisted canonical overlay structure
- overlay arrays preserve safe merge shape
- manifest entries are not orphaned
- manifest source hashes and output hashes still match the current canonical content and overlay files

It does not currently try to judge translation quality. In particular, it will not flag:

- leftover English in translated prose
- awkward or inconsistent HK terminology
- symbol drift that is still structurally valid
- compact marker remnants that happen to survive JSON validation

For that reason, zh-HK rollout verification should be a two-step flow:

1. Use targeted translation/resume to finish the remaining failed concepts.
2. Run `pnpm i18n:validate -- --locale zh-HK` for structural integrity, then audit the translated overlays directly for English leftovers, marker hygiene, and pedagogical coherence.

Additional QA guardrails now exist for the shipped zh-HK product surface:

- `pnpm i18n:check:zh-HK`
  - runs `scripts/check-zhhk-locale-quality.mjs`
  - scans `messages/zh-HK.json`, `content/i18n/zh-HK/**/*.json` shard files, and `content/i18n/generated/zh-HK.json`
  - flags suspicious Simplified Chinese characters, corrupted `????`-style text, and obvious English leftovers in high-risk visible fields
- `pnpm i18n:sweep:zh-HK`
  - runs `scripts/browser-zhhk-site-sweep.mjs`
  - uses Playwright DOM text inspection in a real browser
  - sweeps the primary zh-HK public routes plus the signed-in account/dashboard routes through the existing dev-harness cookie seam
  - writes a JSON report to `output/browser-zhhk-site-sweep.json`

Recommended final-site verification flow:

1. `pnpm i18n:validate -- --locale zh-HK`
2. `pnpm i18n:check:zh-HK`
3. Build the production server and run the public sweep with `OPEN_MODEL_LAB_BASE_URL=<prod-like-local-url> pnpm i18n:sweep:zh-HK`
4. Run the signed-in portion against a dev server with `ENABLE_DEV_ACCOUNT_HARNESS=true` and `OPEN_MODEL_LAB_DEV_BASE_URL=<dev-url>`
5. Patch only the exact message, content-overlay, or shared-component seam that still leaks

## Allowlist And Protected Tokens

The translation pipeline never translates arbitrary JSON keys.

Translated concept fields include visible authored/support text such as:

- `title`
- `shortTitle`
- `summary`
- `highlights`
- `topic`
- `subtopic`
- `recommendedNext[].reasonLabel`
- explanation paragraphs
- key ideas
- misconception text
- worked examples
- mini challenge text
- quick-test text
- accessibility descriptions
- page-framework section titles
- selected equation labels, notes, and variable-link descriptions when they are user-facing

Translated catalog fields include visible labels and summaries such as:

- subject/topic titles and descriptions
- starter-track titles, summaries, introductions, sequence rationale, and checkpoints
- guided-collection titles, summaries, introductions, highlights, steps, and educator notes
- recommended goal-path titles, summaries, objectives, milestones, and next-step notes

Never translate:

- ids
- slugs
- storage/progress/share/analytics keys
- route identifiers
- URLs
- formula tokens
- code-like markers
- placeholders

Token-preservation is strict on counts, not order. If the same protected tokens appear in a different order, that is acceptable. If a protected token is missing or unexpected, the chunk still fails.

Protected token handling lives in `tools/i18n/tokens.py`.

## Glossary, Cache, And Staleness

Locale glossary files live in `tools/i18n/glossaries/<locale>.json`.

- Use them to keep core scientific/product vocabulary stable.
- The mock provider applies the glossary deterministically.
- The Ollama provider receives glossary terms in the prompt and keeps the response JSON-only.

Each locale keeps checked-in manifest state alongside the overlays:

- `content/i18n/<locale>/manifest.json`

`manifest.json` tracks per-shard source hashes, output hashes, attempts, provider, timestamps, and status.

A local ignored `content/i18n/<locale>/.translation-memory.json` cache may be created by translation tooling. It caches exact source-string translations by provider fingerprint plus source-text hash so reruns do not have to retranslate every repeated string, but it is not runtime data and is not part of the public release tree.

This makes `--changed-only` and retry/resume behavior practical for incremental authoring.

## Legacy Migration

Legacy monoliths are still tolerated as migration inputs:

- `content/i18n/<locale>/catalog.json`
- `content/i18n/<locale>/concepts.json`

`scripts/generate-i18n-content-bundle.mjs` will split those monoliths into the preferred shard files when the shards are missing, then regenerate the runtime bundle from the shard tree.

- 5 catalog shard files
- 92 concept shard files

## Finishing Failed Concepts And Auditing Translated Overlays

When a translation wave leaves stragglers, use this order:

1. Retry only the failed or stale concepts with `--resume`.
2. If a concept still fails, regenerate just the stubborn field or the smallest failing chunk rather than rerunning the whole corpus.
3. Run `pnpm i18n:validate -- --locale zh-HK` once the failed set is finished.
4. Audit already-translated overlays for:
   - raw English leftovers
   - malformed or surviving protected markers
   - broken arrays or malformed partial overlays
   - mistranslated symbols, formulas, or pedagogy
5. Patch only the files that still look wrong, then rerun validation.

This repo does not currently rely on a separate quality gate for translated prose. The structural validator is necessary, but not sufficient, so the final human review step still matters for zh-HK concept content.

## Adding A New Locale

1. Add the locale to `i18n/routing.ts`.
2. Add `messages/<locale>.json`.
3. Add `tools/i18n/glossaries/<locale>.json` if you want controlled terminology.
4. Run `pnpm i18n:translate -- --locale <locale> --provider ollama --model <local-model>`.
5. Review the generated `content/i18n/<locale>/...` shards.
6. Run `python tools/i18n/validate_overlays.py --locale <locale>`.
7. Run `pnpm content:registry` so the runtime bundle includes the new locale overlays.
8. Add a locale-quality glossary if needed, then run `pnpm i18n:check:zh-HK` or an equivalent checker for the new locale.

## Human Review Still Required

Machine-assisted overlays are a baseline, not final copy.

Review these areas carefully before shipping a new locale wave:

- scientific terminology consistency
- idiomatic phrasing for learner-facing explanations
- quick-test distractors and explanations
- misconception wording
- accessibility descriptions
- SEO copy
- any equation-adjacent prose where terminology precision matters
- any machine-generated shard that still needs human review
